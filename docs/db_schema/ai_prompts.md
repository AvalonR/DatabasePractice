# AI Prompt Documentation for Data Generation

## Overview
This document records the prompts used to generate the 100+ bulk records per table in `filled_schema.sql` (Section C), along with issues encountered and workarounds applied.

---

## Prompt: Roles (100 records)

**Prompt text:**
> Generate 100 unique role titles for a logistics/delivery company database. Provide them as a comma-separated list suitable for an SQL INSERT statement. Include common titles like "Driver", "Manager", "Dispatcher" spread throughout.

**Issues encountered:**
- The AI initially generated only 40 roles. The prompt had to be refined with "at least 100, exactly 100, no fewer" to reach the required count.
- Some roles were duplicates (e.g., "Supervisor" and "Shift Supervisor" appeared twice). Duplicates were manually removed and replaced with alternate titles.
- Final output was padded with generic roles (e.g., "Cleaner", "Janitor", "Handyman") to reach 100 since realistic logistics roles are limited.

## Prompt: Permissions (100 records)

**Prompt text:**
> Generate 100 unique permission keys for a delivery management system. Use snake_case format like "manage_orders", "view_reports". Group them by domain: fleet, orders, users, finance, settings, security, reports, and integrations.

**Issues encountered:**
- The AI generated permission keys with inconsistent naming (mixed camelCase and snake_case). The output was post-processed to enforce `snake_case` consistently.
- Only about 60 structurally unique permissions exist in a realistic system. Remaining 40 entries were generated by adding prefix-suffix patterns (e.g., `view_archive`, `manage_archive`, `view_recycling`, `manage_recycling`).
- Two permissions named "manage_settings" appeared twice; one instance was renamed to "manage_system_settings".

## Prompt: Map Regions (100 records)

**Prompt text:**
> Generate 100 geographical zone names for a delivery network map, each with a risk level (Low, Medium, or High). Examples: "Central District", "Industrial Zone". Provide as table rows with region_name and risk_level columns. Ensure roughly 50% Low, 30% Medium, 20% High distribution.

**Issues encountered:**
- The AI generated many real-world location names (e.g., "Manhattan", "Brooklyn") which were inappropriate for a fictional system. These were replaced with generic/descriptive names like "North Valley", "East Ridge".
- Risk level distribution was off — the AI produced ~70% Low. Weights were manually adjusted to match the target distribution.
- The AI occasionally included non-geographic entries like "Customer Service Zone" which were replaced with more map-appropriate names.

## Prompt: Nodes (110 records)

**Prompt text:**
> Generate 110 delivery network node locations on a 200x200 Cartesian grid. Each node must have x_coord (10-200), y_coord (10-200), a descriptive label (e.g., "Sorting Center", "Parcel Hub A"), and a map_region_id (1-10). Arrange them in 10 rows of 11 nodes each, with roughly uniform spacing.

**Issues encountered:**
- The AI did not follow the grid layout instruction; coordinates were randomly scattered. The final data was manually generated following a structured grid pattern (10 rows × 11 columns with 10-15 unit spacing).
- The AI generated duplicate labels (e.g., "Warehouse 1" appeared twice). These were deduplicated by appending unique numbers.
- The AI could not consistently format the output as SQL INSERT statements. All 110 rows were manually formatted after generating the coordinate pattern programmatically.

## Prompt: Edges (110 records)

**Prompt text:**
> Generate 110 road connections (edges) between nodes in a delivery network. Each edge connects two nodes (node_a_id, node_b_id) with distance_units (5.0-50.0) and speed_limit (30-70 km/h). Create a connected graph by linking adjacent nodes in a grid pattern.

**Issues encountered:**
- The AI generated edges between non-adjacent nodes, creating a non-planar graph that could not be rendered on a 2D map. All edges were manually defined as horizontal, vertical, and short diagonal connections.
- Many distance values were unrealistic (e.g., 0.5 units for nodes 50 units apart). Distances were recalculated based on actual node coordinates.
- The AI did not ensure graph connectivity — the generated edges formed 4 disconnected subgraphs. The final set was manually verified for full connectivity.

## Prompt: Locations (100 records)

**Prompt text:**
> Generate 100 location records that pair each delivery node with a descriptive name and address. Use the pattern "Location Point {id} - {node_label}" for names and "Address {id}, Zone {region_name}" for addresses.

**Issues encountered:**
- The AI generated locations for non-existent nodes (IDs beyond the actual range). Fixed by using an INSERT...SELECT query that only references existing node IDs.
- Address formatting was inconsistent — some had full addresses, others had only city names. The simplified pattern "Address {id}, Zone {name}" was adopted for uniformity.
- No data quality issues after switching to the SELECT-based approach since the query derives data directly from existing nodes/regions.

## Prompt: Customers (100 records)

**Prompt text:**
> Generate 100 customer records for a delivery system. Each record needs first_name, last_name, email (realistic format), and phone (Lithuanian format +3706XXXXXXXX). Use diverse first and last names from various cultural backgrounds.

**Issues encountered:**
- The AI generated duplicate email addresses (two different customers with the same domain+name pattern). All email uniqueness was verified.
- Phone numbers did not follow the requested Lithuanian format (+3706XXXXXXXX). Some had incorrect prefixes like +1 or +44. All phones were regenerated with the correct format.
- The AI created 100 customers but only produced about 60 unique last names. Additional names were sourced from a secondary generation pass.

## Prompt: Staff (100 records)

**Prompt text:**
> Generate 100 staff members for a logistics company. Include Drivers (at least 40), Warehouse Operators (15), Fleet Supervisors (5), Dispatchers (5), and various other positions (Technicians, Safety Officers, Managers, Administrators, etc.). Provide first_name, last_name, position, and hire_date (between 2024-01-01 and 2024-12-01).

**Issues encountered:**
- The AI generated only 65 staff despite requesting 100. A second pass was needed to generate the remaining 35.
- Position distribution was imbalanced — only 25 Drivers vs the requested 40. The balance was manually adjusted.
- The AI used "N/A" for some hire dates. All dates were replaced with realistic values.
- Several last names were duplicated across different first names. This was accepted as realistic since unrelated staff can share last names.

## Prompt: Users (100 records)

**Prompt text:**
> Generate 100 user accounts for a delivery management system. Each user needs a username (first initial + last name, lowercase), a bcrypt password hash (use the placeholder '$2a$10$HpyiB4n6cVLv0nu6Ta3dbeyjW.PYWo0hoG11OklU27Sy6UeTxiqJS'), a role_id (1-10), and either a staff_id (for employees) or customer_id (for customers). Assign staff_id for roles 2-4,6-10, and customer_id for role 5.

**Issues encountered:**
- The AI generated usernames with spaces and special characters (e.g., "john!smith"). All usernames were manually verified and corrected to match the `[a-z]+` pattern.
- The AI assigned role_id 1 (Admin) to 15 users, which is unrealistic. Admin count was reduced to 2.
- Several users were assigned both a staff_id AND a customer_id, which violates the intended data model. These were split — employee users got staff_id=NULL customer_id=NULL, customer users got customer_id and no staff_id.
- Duplicate usernames occurred when two staff had the same first initial + last name combination. Affected usernames were disambiguated with a numeric suffix.

## Prompt: Vehicle Types (50 + 50 records)

**Prompt text:**
> Generate 50 vehicle types for a delivery fleet. Include a variety: vans, trucks, special-purpose vehicles (refrigerated, tankers, flatbeds), and light vehicles (motorcycles, bicycles, drones). For each type provide: type_name, fuel_rate (0.01-0.65), max_weight_capacity (5-14000 kg), and price_per_kg (2.50-20.00).

**Issues encountered:**
- The AI generated only 28 types. The remaining 22 were created by the developer based on common vehicle categories (e.g., "Mobile Library", "Blood Bank Van", "Art Transport").
- Fuel_rate values were inconsistent — some were clearly wrong (e.g., 0.90 for a bicycle). All values were adjusted to realistic ranges.
- The AI generated vehicle types that were too similar (e.g., "Small Van", "Medium Van", "Large Van", "Extra Large Van"). Differentiation was improved by adding unique features (e.g., "Insulated Van", "Refrigerated Van", "Cargo Van").
- Some weight capacities were physically impossible (e.g., 2,000,000 kg for a standard truck). These were corrected.

## Prompt: Vehicles (100 records), Orders (100), Routes (100), Route Segments (100), Deliveries (100), Maintenance Logs (100)

**Prompt text for vehicles:**
> Generate 100 vehicles assigned to vehicle_types 1-50. Each vehicle needs a unique license_plate ("GEN" + 5-digit zero-padded number) and a current_status distributed as: 60% Available, 20% On Route, 10% Maintenance, 10% Retired.

**Prompt text for orders:**
> Generate 100 orders distributed across customers 1-25 and nodes 11-60. Include order_date spread over the last 200 days, random total_weight (10-1010 kg), and a status distribution: 40% Delivered, 20% In Transit, 20% Pending, 10% Draft, 5% Cancelled, 5% Failed.

**Prompt text for routes:**
> Generate 100 routes assigned to vehicles 11-60 and drivers (staff) 11-50. Use planned_date spread over 150 days, random total_distance (20-220 km), and status: 25% Completed, 15% Cancelled, 10% Active, 50% Planned.

**Prompt text for route_segments:**
> Generate 100 route_segments linking routes 11-110 to edges 11-120 with sequence_order 1-10.

**Prompt text for deliveries:**
> Generate 100 deliveries linking orders 11-110 to routes 11-110. Status distribution: 40% Delivered, 30% Pending, 20% In Transit, 10% Failed. Delivered/Failed entries should have actual_time set.

**Prompt text for maintenance_logs:**
> Generate 100 maintenance_logs for vehicles 11-110. Include service_date spread over the last 300 days, one of 8 predefined descriptions (randomized), and cost between $150-$3,150.

**Issues encountered (all tables):**
- The AI could not generate correctly formatted SQL at scale — output was mixed with explanatory text and markdown. All 100+ row inserts for these tables were generated using programmatic SQL generation (numbered subqueries with `@n` counters) rather than manual INSERT statements.
- The AI's status distributions were inaccurate. When programmatic generation was used instead, the distributions were enforced exactly via `CASE WHEN n MOD x = 0 THEN ...` patterns, which may differ from true randomness but satisfy the minimum record count requirement.
- Foreign key references were a recurring issue — the AI frequently used IDs that didn't exist in parent tables (e.g., customer_id=999 when only 100 customers existed). The programmatic approach with modular arithmetic (`(n % 25) + 1`) guarantees all FK references are valid.
- Date generation was inconsistent — the AI used past, present, and occasionally future dates. The programmatic approach uses `DATE_SUB(CURDATE(), ...)` for consistent past dates.

**Total records:** ~1,795 AI-generated + 180 Section A + ~100 Section B = ~2,075

**Key lesson:** AI excels at generating varied text data (names, descriptions, labels) but performs poorly at generating structured relational data with correct foreign key references. The most effective workflow was: (1) use AI for domain-specific text content, (2) use programmatic SQL generation for bulk numeric data with valid FK references.
